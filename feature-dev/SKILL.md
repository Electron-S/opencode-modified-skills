# 피처 개발 워크플로우

새 기능을 체계적으로 구현하는 7단계 접근법. 코드 작성 전에 코드베이스를 이해하고, 모호함을 해소하고, 아키텍처를 설계하는 것이 핵심이다.

## 핵심 원칙

- **먼저 이해, 그 다음 행동**: 기존 코드 패턴을 먼저 읽고 파악한다
- **에이전트가 반환한 파일을 읽는다**: 에이전트가 중요하다고 식별한 파일은 직접 읽어 깊은 컨텍스트를 쌓는다
- **단순하고 우아하게**: 읽기 쉽고 유지보수 가능한 아키텍처 우선
- **TodoWrite 활용**: 7단계 전체 진행 상황을 추적한다

---

## 1단계: 발견 (Discovery)

**목표**: 무엇을 만들어야 하는지 이해한다

**액션**:
1. TodoWrite로 7단계 전체 할 일 목록 생성
2. 기능 요청이 불명확하면 다음을 질문:
   - 어떤 문제를 해결하는가?
   - 기능이 무엇을 해야 하는가?
   - 제약 조건이나 요구사항이 있는가?
3. 이해한 내용을 요약하고 유저에게 확인

---

## 2단계: 코드베이스 탐색 (Codebase Exploration)

**목표**: 관련 기존 코드와 패턴을 이해한다

**액션**:
1. `code-explorer` 에이전트 2~3개를 **병렬로** 실행. 각 에이전트는 다른 측면에 집중:
   - "유사한 기능을 찾고 구현 흐름을 추적한다"
   - "[기능 영역]의 아키텍처와 추상화를 매핑한다"
   - "[관련 기능]의 현재 구현을 분석한다"
   - "UI 패턴, 테스트 방식, 또는 [기능]에 관련된 확장 포인트를 파악한다"

2. 에이전트 실행 방법: Agent 툴을 사용해 각 에이전트를 병렬로 실행하거나, [./agents/code-explorer.md](./agents/code-explorer.md)의 지침을 따라 직접 탐색 수행

3. 에이전트가 식별한 **핵심 파일 5~10개를 직접 읽어** 깊은 이해 구축

4. 발견한 패턴과 인사이트를 종합 정리해서 유저에게 제시

---

## 3단계: 명확화 질문 (Clarifying Questions)

**목표**: 아키텍처 설계 전에 모든 모호함을 해소한다

**중요**: 이 단계는 절대 건너뛰지 않는다.

**액션**:
1. 코드베이스 탐색 결과와 원래 기능 요청을 검토
2. 명세되지 않은 부분을 식별:
   - 엣지 케이스
   - 에러 처리
   - 통합 포인트
   - 범위 경계
   - 설계 선호도
   - 하위 호환성
   - 성능 요구사항
3. **질문을 정리된 목록으로 유저에게 제시**
4. **답변을 받기 전까지 아키텍처 설계로 넘어가지 않는다**

유저가 "알아서 해줘"라고 하면, 권장 방향을 먼저 제시하고 명시적 확인을 받는다.

---

## 4단계: 아키텍처 설계 (Architecture Design)

**목표**: 다양한 구현 방식을 설계하고 최선을 선택한다

**액션**:
1. `code-architect` 에이전트 2~3개를 **병렬로** 실행, 각각 다른 관점:
   - **최소 변경**: 최소한의 변경, 최대한의 재사용
   - **클린 아키텍처**: 유지보수성, 우아한 추상화
   - **실용적 균형**: 속도 + 품질

2. 에이전트 실행 방법: Agent 툴을 사용해 병렬 실행하거나, [./agents/code-architect.md](./agents/code-architect.md)의 지침을 따라 직접 설계

3. 모든 방식을 검토하고 이 태스크에 가장 적합한 방식에 대한 의견을 형성

4. 유저에게 다음을 제시:
   - 각 방식의 간략한 요약
   - 트레이드오프 비교
   - **추론과 함께 권장 방식**
   - 구체적인 구현 차이점

5. **유저에게 어떤 방식을 선호하는지 질문**

---

## 5단계: 구현 (Implementation)

**목표**: 기능을 만든다

**유저 승인 없이 시작하지 않는다.**

**액션**:
1. 명시적 유저 승인 대기
2. 이전 단계에서 식별된 모든 관련 파일 읽기
3. 선택한 아키텍처를 따라 구현
4. 코드베이스 컨벤션을 엄격히 따름
5. 깔끔하고 잘 구조화된 코드 작성
6. 진행하면서 TodoWrite로 할 일 업데이트

---

## 6단계: 품질 리뷰 (Quality Review)

**목표**: 코드가 단순하고, 중복 없고, 우아하며, 기능적으로 올바른지 확인한다

**액션**:
1. `code-reviewer` 에이전트 3개를 **병렬로** 실행, 각각 다른 초점:
   - 단순성/DRY/우아함
   - 버그/기능 정확성
   - 프로젝트 컨벤션/추상화

2. 에이전트 실행 방법: Agent 툴을 사용해 병렬 실행하거나, [./agents/code-reviewer.md](./agents/code-reviewer.md)의 지침을 따라 직접 리뷰

3. 결과를 통합하고 수정을 권장하는 가장 심각한 이슈 식별

4. **결과를 유저에게 제시하고 무엇을 할지 질문**:
   - 지금 수정
   - 나중에 수정
   - 그대로 진행

5. 유저 결정에 따라 이슈 처리

---

## 7단계: 요약 (Summary)

**목표**: 완료된 내용을 문서화한다

**액션**:
1. 모든 할 일을 완료로 표시
2. 다음을 요약:
   - 무엇을 만들었는가
   - 핵심 결정 사항
   - 수정된 파일 목록
   - 제안된 다음 단계

---

## 언제 사용하나

**사용할 때:**
- 여러 파일에 걸친 새 기능
- 아키텍처 결정이 필요한 기능
- 기존 코드와의 복잡한 통합
- 요구사항이 다소 불명확한 기능

**사용하지 말 것:**
- 한 줄짜리 버그 수정
- 단순하고 명확한 변경
- 긴급 핫픽스
