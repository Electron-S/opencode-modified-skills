# MCP 서버 평가 가이드

## 평가 목적

MCP 서버의 품질은 구현이 얼마나 잘 돼 있는가가 아니라, **LLM이 MCP 서버만 사용해서 현실적이고 어려운 질문에 얼마나 효과적으로 답할 수 있는가**로 측정한다.

---

## 좋은 평가 질문의 요건

### 필수 특성

| 특성 | 설명 |
|------|------|
| **읽기 전용** | 시스템 상태를 수정하지 않는 비파괴적 작업만 사용 |
| **독립적** | 다른 질문에 의존하지 않음 |
| **비파괴적** | 생성, 수정, 삭제 작업 없음 |
| **복잡함** | 수십 개의 툴 호출과 여러 데이터 소스에 걸친 심층 탐색 필요 |
| **현실적** | 사람이 실제로 관심 가질 사용 사례 기반 |
| **검증 가능** | 직접 문자열 비교로 확인 가능한 단일하고 명확한 답변 |
| **안정적** | 시간이 지나도 답변이 변하지 않는 닫힌 개념 기반 |

### 좋은 답변 형식

- 사람이 읽을 수 있는 형식 선호 (이름, 날짜, 사용자명)
- 불투명한 식별자보다 구체적인 값
- 목록이나 복잡한 구조보다 단일 값
- 숫자, 타임스탬프, 파일명, URL 등 다양한 형태 가능

### 피해야 할 것

- 동적으로 변하는 데이터 (예: "이 게시물의 반응 수")
- MCP 서버 구현 코드 자체를 읽는 것
- 생성, 수정, 삭제가 필요한 작업
- 단순 키워드 검색으로 답할 수 있는 질문

---

## 평가 생성 프로세스

다음 5단계를 순서대로 진행한다:

### 1단계: 문서 검사
- API 문서와 MCP 서버 README 검토
- 제공하는 데이터 유형과 사용 사례 파악

### 2단계: 툴 검사
- 사용 가능한 모든 툴 목록 확인
- 각 툴의 파라미터와 반환 값 파악
- 툴들이 어떻게 조합될 수 있는지 이해

### 3단계: 이해 개발
- 탐색적 쿼리로 실제 데이터 구조 파악
- 서버 기능과 한계 확인

### 4단계: 읽기 전용 콘텐츠 탐색
- 읽기 전용 작업으로만 데이터 탐색
- 흥미롭고 비자명적인 데이터 패턴 찾기
- 작은 타깃 도구 호출로 점진적으로 탐색

### 5단계: 태스크 생성
- 탐색에서 발견한 내용 기반으로 질문 작성
- 각 질문을 직접 풀어 답변 확인
- 답변이 안정적이고 검증 가능한지 확인

## 출력 형식

다음 구조의 XML 파일로 저장:

```xml
<evaluation>
  <qa_pair>
    <question>
      In the project management system, there's a task that was originally assigned to 
      one person but later reassigned. The reassigned task is in the "Infrastructure" 
      category and was completed in Q3 2023. Who was the original assignee?
    </question>
    <answer>Alice Johnson</answer>
  </qa_pair>

  <qa_pair>
    <question>
      Find the API endpoint that was deprecated in version 2.1.0 but is still documented 
      in the changelog. What is the recommended replacement endpoint path?
    </question>
    <answer>/api/v2/users/profile</answer>
  </qa_pair>

  <!-- 총 10개의 qa_pair 작성 -->
</evaluation>
```

## 평가 질문 예시

### 좋은 예 (복잡하고 검증 가능)
```
AI 모델 출시 토론에서 동물 코드명이 붙은 모델 중 안전 등급이 ASL-X 형식으로 
결정 중인 것이 있었다. 점박이 야생 고양이 이름의 모델에 대해 결정 중이던 X는 몇이었나?
```
**좋은 이유**: 여러 툴 호출 필요, 단일 검증 가능한 숫자, 안정적인 역사적 데이터

### 나쁜 예 (너무 단순)
```
프로젝트에 몇 명의 사용자가 있나?
```
**나쁜 이유**: 단순 카운트 쿼리, 시간에 따라 변할 수 있음

### 나쁜 예 (동적 데이터)
```
가장 최근에 업로드된 파일은 무엇인가?
```
**나쁜 이유**: 새 파일이 업로드될 때마다 답변이 변함

## 평가 실행

평가는 stdio, SSE, HTTP 세 가지 전송 방식을 지원하는 스크립트로 실행할 수 있다. 실행 결과 보고서에는 다음이 포함된다:

- 전체 정확도 지표
- 툴 사용 통계
- 태스크별 결과 분석
- 개선이 필요한 영역 식별

---

## 체크리스트

평가 제출 전 확인:

- [ ] 정확히 10개의 qa_pair 작성
- [ ] 모든 질문이 읽기 전용 작업만 필요
- [ ] 각 질문의 답변을 직접 확인
- [ ] 답변이 시간이 지나도 변하지 않는 안정적인 데이터
- [ ] 질문이 서로 독립적
- [ ] 답변이 직접 문자열 비교로 검증 가능
- [ ] 각 질문이 여러 툴 호출을 필요로 하는 복잡한 탐색 요구
